## 1. How the project evolved

The 10 days break into three overlapping campaigns: a **UX polish burst**, a
**scoring-correctness firefight**, and a **cost/invocation cleanup**.

### A. WC surface UX polish (peak: Jun 12)

The bulk of Jun 12's 22 commits were design iteration on the `/wc` surface —
the only product surface real users touch:

- **Finished-fixture cards** redesigned repeatedly for a score-centered,
  symmetrical layout with correctness rings (`364c5fa`, `d423d44`, `f6852e1`,
  `ca64c7d`, `9c01329`, `f457973`).
- **TabBar** redesign — larger icons, background pill, press feedback, iOS
  overscroll padding, mobile touch responsivity (`00ae5e2`, `b00e7f3`,
  `9a1eda2`).
- **Chat** consolidated into a dedicated `/wc/chat` page with result
  notifications, auto-grow composer, date separators, long-press action sheet
  (`37e7b91`, `1aaf07a`, `345fc05`).
- **Results tab** — concluded fixtures behind a toggle, persisted show/hide
  state in localStorage, rivals CTA gating (`4db0c1e`, `4f1438c`, `dbc0d98`).
- **Misc legibility** — self-hosted WC nation flags (`389fd40`), score-input
  contrast for outdoor viewing (`3735ae9`), thicker correctness rings,
  systematic touch-target padding (`cccfaa9`).

### B. Scoring & results correctness firefight (Jun 13 → 17)

The hardest, highest-stakes work. A live tournament means scoring bugs cost
real users real points, so these were urgent:

- **Leaderboard showed zero points** — Supabase/PostgREST silently caps
  results at 1,000 rows. With ~50 users × ~40 scored prediction-types the
  query truncated and many users summed to zero. Three commits chased it:
  a `.limit(10000)` band-aid (`b1b8282`, which *cannot* work — a client limit
  can't exceed the server max-rows cap), then the real fix: a
  `sum_prediction_points()` SQL RPC doing `SUM/GROUP BY` server-side
  (`b1b8282` → `73f5d07` → `11a5abf`).
- **Draw scoring** — providers emit `winner: null` for draws and the EPT
  config's `allow_draw` flag was often absent, so drawn soccer matches voided
  every prediction. Fixed by inferring `allow_draw` from the options list and
  normalising provider draw output (`9dd4601`).
- **Prediction contradiction race** — the browser sent two winner writes per
  score save; the redundant client write arrived last and overwrote the
  server's correct auto-derivation, costing users points. Fixed by making the
  server the sole writer (`60d4ef0`).
- **Scoring hardening** — transactional updates, `result_confirmed` filter,
  auto-refresh (`bf8c9f7`).
- **Result propagation & automation** — confirmed results now propagate to all
  competition instances sharing a fixture (`f727b15`); pg_net's 5s timeout was
  silently killing every results cron run (`a40be6d`); ESPN search fixes for
  UTC-midnight events and non-default `provider_league` slugs (`30d607f`,
  `96a76e3`, `0634c35`, `7325cf4`); sport-specific live-window failsafe
  replacing a hardcoded 6h (`09e857b`).

### C. Notifications & cost (Jun 12, 17)

- **Notification spam** fixed with daily-tag dedup and de-duplicated result
  checks (`dc2d49f`, `ae0610b`); smart reminders only for incomplete
  predictions, user-configurable, per-competition mute (`7a1a9e6`, `0dfaf36`).
- **Vercel invocation optimization (Jun 17, this campaign)** — see §2.

### Through-line

The product is past "build features" and deep into **operating a live system
under free-tier constraints**: data-integrity bugs that only surface at real
scale (row caps, races, provider quirks), and resource bugs that only surface
under real traffic (invocation loops). Static review alone missed both classes
— production data (DB row counts, runtime logs) was what exposed them.

## 2. The invocation optimization (Jun 17)

Triggered by a Vercel alert at 75% of the 1M monthly function-invocation free
tier. Root causes found and fixed:

1. **Dashboard result-check polling** (`DashboardClient.tsx`) looped one
   `POST /api/results/check` per eligible event every 5 min, regardless of tab
   visibility → gated to a single backstop event, every 15 min, visible-tab
   only (`91196c1`).
2. **Live dashboard refresh** — `router.refresh()` every 60s re-rendered the
   force-dynamic page for every viewer → 180s, visible-tab only (`91196c1`).
3. **Infinite `my-group` refetch loop** (`ClassificationTabs.tsx`) — the
   single largest source. `visibleClassifications` was a `.filter()` computed
   inline (new array reference every render) used as an effect dependency, so
   on the groups tab each fetch → `setGroupData` → re-render → new reference →
   refetch, looping at the ~1.5–2s network round-trip. **Production runtime
   logs showed ~40 `/api/tournament/my-group` calls per minute per open
   leaderboard tab.** Fixed with `useMemo` (`98427fe`). Post-deploy logs
   confirmed: ~2 calls per 20 min across many leaderboard loads, no errors.

Key reframe: **Vercel function invocations ≠ Supabase queries.** A dynamic
page render is one invocation no matter how many DB queries it runs. The cost
driver is the *number of requests to functions* — dominated by client-side
polling and render-driven refetch loops, not query counts.

## 3. What Claude can `/learn` from this

Concrete, recurring footguns from these 10 days. **Proposed additions to
`CLAUDE.md`** (project memory) so future local Claude Code sessions catch them
early. Several reinforce gotchas already documented — the repetition is the
signal.

### 3.1 React: never put inline-computed arrays/objects in effect deps

> A `.filter()`/`.map()`/object literal computed in a component body is a **new
> reference every render**. If it's in a `useEffect`/`useMemo`/`useCallback`
> dependency array that also triggers a state update, you get an infinite
> loop throttled only by latency. Wrap such values in `useMemo` before using
> them as dependencies. This caused a ~40 req/min `/api/tournament/my-group`
> loop (`98427fe`). When reviewing any `useEffect` that fetches + setsState,
> check every dependency for reference stability.

### 3.2 Vercel cost: count requests, not queries

> On Vercel, **function invocations are requests to functions**, not DB
> queries — a dynamic page render is one invocation regardless of internal
> query count. The dominant cost is **client-side polling and refetch loops**.
> When auditing invocations, hunt for: `setInterval` refetch loops,
> `router.refresh()` on timers, render-driven refetch loops (see 3.1), and
> over-broad `force-dynamic`. Gate any polling with: visibility checks
> (`document.visibilityState`), the longest acceptable interval, and
> single-item rather than per-item fan-out.

### 3.3 Don't add per-user client polling as a "reliability backstop" without gating

> `b377aa9` added a 5-min, all-events, always-on client result-check loop as a
> safety net; `91196c1` had to walk it back because it multiplied invocations
> across every viewer. A backstop is fine, but make it cheap by default:
> visible-only, long interval, single item. Prefer server cron as the primary
> mechanism; the client should be a thin fallback.

### 3.4 PostgREST silently caps at 1,000 rows — aggregate in SQL (reinforce existing gotcha)

> Already in `CLAUDE.md`, but it cost three commits (`b1b8282`, `73f5d07`,
> `11a5abf`) and zeroed the live leaderboard. A client `.limit(10000)` **cannot**
> override the server-side `PGRST_DB_MAX_ROWS` cap. Any cross-user/event
> aggregation must use a `SUM/GROUP BY` RPC, never a JS reduce over fetched
> rows. Treat unbounded `.select()` on prediction-scale tables as a bug.

### 3.5 Trust production telemetry over static analysis for scale/perf bugs

> Both the row-cap zeroing and the refetch loop were **invisible in code
> review** and obvious in production data (row counts; Vercel runtime logs).
> For any perf/cost/scale investigation, pull the live signal first
> (`get_runtime_logs`, DB counts) — it points straight at the hot path and
> prevents inflated guesses. After a perf fix, re-pull logs to confirm the
> behavior actually changed.

### 3.6 Sports-provider data is inconsistent — normalize defensively

> Recurring provider quirks bit scoring repeatedly: draws emitted as
> `winner: null` (`9dd4601`), events near UTC midnight missed by date ranges
> (`30d607f`), non-default league slugs ignored (`96a76e3`, `0634c35`), and
> pg_net's 5s timeout killing cron calls (`a40be6d`). When touching provider
> code, assume nulls, timezone edges, and per-league config divergence.

### 3.7 One writer per field — avoid client+server double-writes

> The contradiction race (`60d4ef0`) came from both the client and the
> server's auto-derive writing the winner field, with the later (client) write
> overwriting the correct one. Pick a single source of truth for each derived
> field; don't have the client redundantly POST something the server already
> computes.

### Optional: encode 3.1–3.3 as a lightweight guardrail

For the local setup, consider a `PostToolUse`/`Stop` hook or a CLAUDE.md
checklist item that flags new `useEffect` blocks containing both a `fetch` and
a non-primitive dependency, and any new `setInterval`/`router.refresh` on a
timer — the three patterns most likely to silently burn invocations.
